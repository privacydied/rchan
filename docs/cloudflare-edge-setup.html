Runbook · Cloudflare + rchan

Six edge changes to finish the perf pass

The code side is deployed — short-TTL headers, prerender hints, background-poll throttling and the offline page are live. These Cloudflare dashboard changes are what let the edge actually act on them. Work top to bottom: the toggles are zero-risk, the cache rules get more consequential as you go.

Run every Verify command from your laptop (not the NAS — it only reaches the origin on localhost). The signal you're always looking for is the cf-cache-status response header; its values are decoded in the legend at the bottom.
Current status — rules 01–05 are live. Rule 06 (anonymous HTML caching) is deliberately not enabled: the staleness + staff-bypass risk isn't worth it here. It's kept documented below in case that calculus ever changes.

01

Confirm Brotli is compressing your statics

Impact · Medium Risk · None Verify only

Why

Your origin only gzips (level 6, no Brotli module). Cloudflare can Brotli-recompress cacheable text at the edge, shaving ~15–20% off the 145 KB ux.css + 250 KB ux.js first load for zero origin change. This is almost always already on — you're just confirming.

Dashboard

Speed›Optimization›Content Optimization›Brotli

Ensure it's On. On newer dashboards compression is automatic or lives under Rules › Compression Rules — if you don't see the toggle, the Verify below is the real test.

Verify

# from your laptop
curl -sI -H 'Accept-Encoding: br' \
  https://boards.rchan.xyz/.rchan/ux.css?v=c5f84671 \
  | grep -i content-encoding
# expect → content-encoding: br

If it says gzip, flip Brotli on and re-run.

02

Turn on Smart Tiered Cache

Impact · Medium Risk · None One toggle

Why

By default every Cloudflare data centre fills its cache straight from your origin — so a popular thumbnail can be pulled from the NAS once per region. Smart Tiered Cache funnels all POPs through a single upper tier first, so your residential-grade uplink answers each asset roughly once, not dozens of times.

Dashboard

Caching›Tiered Cache›Smart Tiered Cache Topology

Set to On. Nothing else to configure.

Verify

No direct header for this one — its effect shows up as your origin's nginx access log thinning out for /.media/ once Rule 04 is also live. Safe to enable and move on.

03

Turn on Early Hints

Impact · Low–Med Risk · None One toggle

Why

Your origin already emits a Link: rel=preload header for the two render-blocking injected assets (ux.css + predark.js). Early Hints replays that as an HTTP 103 before the origin even finishes responding, so the browser starts fetching them earlier — a real first-paint win that costs nothing because the header already exists.

Dashboard

Speed›Optimization›Content Optimization›Early Hints

Set to On.

Verify

A 103 is hard to see in curl -I. Open a thread in Chrome DevTools → Network → click the document → the Timing tab shows an "Early Hints" phase once it's working. Optional and cosmetic — trust the toggle.

04

Cache thumbnails & media at the edge

Impact · High Risk · Low Cache Rule

Why

Your origin already says max-age=2592000, immutable (30 days) on /.media/, but Cloudflare caches by file extension by default — and your thumb URLs are extensionless (/.media/t_<hash>), so they're treated as dynamic and re-pulled from GridFS on every view. This one rule is your single biggest origin-bandwidth lever.

Dashboard

Caching›Cache Rules›Create rule

When incoming requests match

starts_with(http.request.uri.path, "/.media/")

Then

Cache statusEligible for cache
Edge TTLUse cache-control header if present — respects your 30-day immutable
Browser TTLRespect origin

Verify

# grab a real thumb path from a catalog page's <img src>, then:
curl -sI https://boards.rchan.xyz/.media/t_<hash> \
  | grep -i cf-cache-status
# 1st → MISS   2nd → HIT

With a 30-day edge cache, a media takedown needs a manual purge — see Purge at the bottom. That 30-day window is the deliberate trade for the bandwidth saving.

05

Cache catalog & thread JSON for 15s

Impact · High Risk · Low Cache Rule

Why

Your origin sends max-age=0, s-maxage=15 on catalog.json and res/<id>.json — an "edge may hold this 15s" instruction Cloudflare ignores unless a rule marks the path cacheable. Without it, every open tab's auto-refresh, the liveness pill and the watcher poll hit LynxChan directly. This is precisely what makes the new background-tab poll-throttling pay off.

Dashboard

Caching›Cache Rules›Create rule

When incoming requests match (plan-safe, no regex)

(ends_with(http.request.uri.path, "/catalog.json"))
or
(http.request.uri.path contains "/res/"
 and ends_with(http.request.uri.path, ".json"))

Then

Cache statusEligible for cache
Edge TTLUse cache-control header if present — honors s-maxage=15
Browser TTLRespect origin — stays 0, browsers still revalidate

Verify

for i in 1 2; do
  curl -sI https://boards.rchan.xyz/gen/catalog.json \
    | grep -i cf-cache-status
  sleep 1
done
# 2nd (within 15s) → HIT  then cycles HIT→EXPIRED every 15s

If you see DYNAMIC, the rule isn't matching — recheck the expression.

06

Edge-cache anonymous page HTML

Not enabled — by choice Impact · High · structural Risk · Handle with care Cache Rule

Why

Board, catalog and thread HTML is identical for every logged-out visitor — all personalization ((You), watched, unread, theme) is client-side localStorage. Caching it ~20s at the edge turns a burst of 50 readers on a hot thread into roughly one origin render, and makes the hover-prerender almost free at origin. Live updates still arrive over the WebSocket, so ≤20s-old initial HTML is invisible in practice.

Dashboard

Caching›Cache Rules›Create rule

Place this rule below Rules 04 and 05 — their matches are disjoint, but HTML should be the last word.

When incoming requests match

(http.host eq "boards.rchan.xyz")
and (http.request.method eq "GET")
and (not http.cookie contains "hash=")
and (not starts_with(http.request.uri.path, "/."))
and (not ends_with(http.request.uri.path, ".json"))
and (not ends_with(http.request.uri.path, ".js"))

Then

Cache statusEligible for cache
Edge TTLOverride to 20 seconds — the engine marks HTML no-cache, so you must override to cache at all
Browser TTLBypass / no cache — browsers must never hold stale HTML

The safety hinges on one clause: not http.cookie contains "hash=". hash is LynxChan's login-session cookie, so logged-in staff always BYPASS — they see their mod tools, and their rendered pages never get cached for anonymous visitors. Anonymous users have no hash cookie, and the client-side rchan_view preference cookie is deliberately not matched.

Optional extra hardening

Append to also skip auth/mod surfaces outright:

and (not starts_with(http.request.uri.path, "/account"))
and (not starts_with(http.request.uri.path, "/login"))
and (not starts_with(http.request.uri.path, "/mod"))

Verify it works — and that it's safe

  1. Logged out — curl -sI https://boards.rchan.xyz/gen/catalog | grep -i cf-cache-status twice → 2nd is HIT.
  2. As a session — curl -sI --cookie "hash=x; login=you" https://boards.rchan.xyz/gen/catalog | grep -i cf-cache-status → must be BYPASS. Any request carrying a hash cookie that ever returns HIT means the bypass clause is wrong — remove the rule immediately.
  3. Real browser, logged in — open a thread and confirm your delete / ban / IP controls still render.

Trade-off to accept: a brand-new post can be up to 20s late for someone loading the page fresh (the poster sees it instantly; everyone already in the thread gets it over the WebSocket). Tune the Edge TTL down to 10s for snappier, up to 30s for more offload.

Reading cf-cache-status

HITServed from the edge, origin untouched. The goal state for media, JSON and anon HTML.
MISSNot in cache; fetched from origin and stored (if eligible). The second request should HIT.
EXPIREDWas cached, TTL lapsed, just revalidated. Normal to see this cycle on the 15s JSON rule.
DYNAMICCloudflare chose not to cache — no rule made it eligible. Seeing this on media or JSON means your rule isn't matching.
BYPASSA rule or cookie explicitly skipped cache. The expected, correct result for any logged-in hash request.

Purging

Media takedown — Caching › Configuration › Purge Cache › Custom purge › by URL, and paste the full https:// URL of the file and its t_ thumb. Required because of the 30-day edge cache from Rule 04.

Front-end deploys — no purge ever needed. The ?v= content hash changes the URL on every deploy-fe.sh, so a new bundle is a new cache key.

Emergency — Purge Everything forces a cold edge cache and a brief origin load spike. Reach for it only if something's serving stale that shouldn't be.
